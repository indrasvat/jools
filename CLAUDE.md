# Claude guide to Jools

Jools is a SwiftUI iOS client for Google's Jules cloud coding agent.
This file is the entry point for anyone (including you, Claude)
doing meaningful work in this repo.

**Read this top section always. Drill into the detail sections only
when a task touches them.**

---

## The essentials (always read)

**Stack:** SwiftUI, Swift 6 strict concurrency, SwiftData, iOS 26+,
Xcode 26.1+. Project is generated by XcodeGen from `project.yml`.

**Core commands:**

```bash
make build      # build for simulator (debug)
make test       # JoolsKit + iOS app tests
make lint       # SwiftLint
make sim-run    # build + install + launch on iPhone 17 Pro
make generate   # regenerate Jools.xcodeproj from project.yml
```

**Before working on visual/timing-sensitive code, read
[`docs/LEARNINGS.md`](docs/LEARNINGS.md).** It contains the hard
lessons from earlier sessions: SwiftUI performance traps, animation
recipes, CI gotchas, simulator hygiene. Checking it first prevents
re-paying tuition.

**At the end of every session, add any new non-obvious lessons to
`docs/LEARNINGS.md`.** That's how the knowledge compounds. Terse
entries are better than verbose ones.

**Commit-and-push discipline:** never commit unless explicitly
asked. Never force-push to `main`. Never `rm -rf` DerivedData or
the keychain without asking. Match the scope of your actions to
what was actually requested — no drive-by refactors.

## Verification protocol: Jules web UI is the reference

The Jules web UI (`jules.google.com`) is the ground truth for how
sessions, activities, plans, and completion data should look. Every
feature or fix to Jataayu's session surface **must** be verified
against the web UI for the same session.

**Tools:**

- **AXe CLI** (`axe`) — drives the iOS simulator: tap, swipe,
  screenshot, describe-ui. Use for navigating the app, taking
  verification screenshots, and confirming tap targets.
- **Chrome DevTools MCP** — connects to Chrome via autoConnect.
  Use to inspect the Jules web UI: scrape rendered DOM content,
  extract Angular component state, intercept button actions
  (`window.open` patching), and compare what the web UI shows
  against what Jataayu shows.

**Protocol for feature/fix work on the session surface:**

1. **Open the same session** in both Jataayu (simulator) and the
   Jules web UI (Chrome). Use Chrome DevTools MCP to inspect what
   the web UI renders — DOM elements, CSS classes, data bindings.
2. **Compare every visible element.** Progress events, plan steps,
   completion card, PR link, diff stats, timestamps, expandable
   sections. If the web UI shows it and Jataayu doesn't, that's a
   gap to investigate.
3. **When the REST API falls short**, document it. The web UI uses
   Google's internal batch RPC (`batchexecute`) which returns
   richer data than the public REST API. Known gaps:
   - Runtime duration is server-computed (we approximate with `~`)
   - `changeSet` artifacts on progress events lack patch content
     (file names not extractable for "Updated X, Y, Z" display)
   - `outputs` array ordering is not guaranteed (search all, never
     `.first`)
4. **Screenshot every state** after a fix with `axe screenshot` and
   visually compare to the web UI. Don't claim "verified" from
   code review alone.
5. **If a feature can't be replicated** due to REST API limitations,
   note it explicitly in the PR/commit and in `docs/LEARNINGS.md`
   rather than silently omitting it.

## When touching the chat surface

<details>
<summary>Layout, snapshots, rendering, freeze prevention</summary>

The chat surface (`Jools/Features/Chat/`) uses a **snapshot
architecture** that is load-bearing for performance:

1. **`ActivitySnapshot` is a value-type `Sendable` struct** holding
   everything the view layer needs. The view layer **never** touches
   the SwiftData `ActivityEntity` directly — if you find yourself
   adding `@Query` or `@PersistedProperty` access inside a view
   body, stop and extend the snapshot instead.

2. **`ActivitySnapshotBuilder` is the only place that reads
   `@PersistedProperty`.** Runs `@MainActor`, produces plain-value
   output. If you need a new field in a snapshot (e.g. parsed
   diff files), parse it once in the builder.

3. **The chat list is a SwiftUI `List`, not `LazyVStack`.** Don't
   change this. List gives you `UICollectionView` cell recycling;
   LazyVStack walks every visible cell's layout on every state
   invalidation. This single change was the primary chat-freeze
   fix.

4. **Markdown in agent messages is flattened.**
   `FlatMarkdownRenderer.render(_:)` produces a small flat array of
   `MarkdownSegment` values (text runs packed into a single
   `AttributedString`, code blocks + tables + thematic breaks as
   their own segments). A typical agent message is 4 segments,
   not 50 nested SwiftUI views. If you see per-inline-node SwiftUI
   views creeping back in, push back — the web UI reference does
   ~3 flat HTML elements per block and that's the target.

5. **Observation surface:** `ChatViewModel` is `@Observable` (Swift
   5.9+ macro, not `ObservableObject`). This gives per-property
   tracking so SwiftUI can skip re-rendering for unrelated state
   changes. Don't mix in `@Published` or `@StateObject`.

6. **Polling drives state, not the chat list.** `PollingService`'s
   `isPolling` flag flips true for ~300ms during each network
   request and false otherwise. UI that needs to reflect "the
   session is working" should gate on the session STATE (e.g.
   `config.showSpinner`), not on `isPolling` (which produces a
   flash-on / gap / flash-on pattern).

**If you break a chat freeze perf fix, the symptoms look like:**
- Sustained scroll + polling triggers beachballs / frozen UI
- `mach_msg_overwrite` dominates main-thread samples
- `ViewGraphHostUpdate.dispatchImmediately` in the backtrace
- Per-message markdown takes > 10ms to render

See `docs/Chat_Freeze_Analysis_and_UIKit_Migration_Plan.md` for the
full forensic history.

</details>

## When touching animations

<details>
<summary>Avatar bob, banner progress strip, motion reference</summary>

**The in-bubble mascot bob (`JulesAvatarView`)** uses
`TimelineView(.animation) + sin()`, calibrated against frame-by-frame
analysis of the Jules web UI: ±2pt amplitude, 2.4s period, pure
sine wave. This pattern is mandatory — `@State + .animation(
.repeatForever)` has state-reset quirks on parent re-render and
easeInOut pauses at the endpoints. See the `bobPeriod` /
`bobAmplitude` constants at the top of the view.

**The banner indeterminate progress strip
(`IndeterminateProgressStrip`)** is linear sawtooth, two bands at
50% phase offset, 2.0s cycle. This is the CSS
`animation: slide Xs linear infinite` pattern (GitHub, Linear,
Safari, Material Design). Autoreverse-sine patterns create
zero-velocity pauses at each edge — don't do that. Don't gate on
`isPolling` — gate on session state.

**When matching a reference animation**, extract frames with ffmpeg
and measure amplitude + period before writing code. See the
"Measure the reference, don't guess" section of
`docs/LEARNINGS.md` for the `tile` filter waterfall recipe.

</details>

## When touching the release pipeline

<details>
<summary>Tagging, CHANGELOG, GitHub Actions workflow</summary>

**Release tags are the trigger.** Push `vX.Y.Z` (annotated tag) and
`.github/workflows/release.yml` handles the rest: validates the
tag, extracts the matching `CHANGELOG.md` section, builds a
Release-config simulator `.app`, zips it, creates a GitHub Release
with the notes + asset.

**The `make release VERSION=X.Y.Z` target** prepares the working
tree: bumps `MARKETING_VERSION` in `project.yml`, regenerates
Jools.xcodeproj, rolls `[Unreleased]` in `CHANGELOG.md` into a
dated `[X.Y.Z]` section. It does NOT auto-commit, auto-tag, or
auto-push — tagging is destructive and stays under human control.
The release commit must include all three changed files:
`git add project.yml CHANGELOG.md Jataayu.xcodeproj/project.pbxproj`.

**All shell logic lives in `scripts/ci-release`**, not in YAML
`run:` blocks. It has subcommand dispatch: `parse`, `verify`,
`notes`, `build`, `package`, `publish`. Each is independently
testable on a dev machine (the first five are — `publish` needs
`GH_TOKEN` + a real release). If you add a step, add a subcommand
and shellcheck it.

**Pre-release tags** (`v1.0.0-alpha.1`): the numeric portion is
compared against `MARKETING_VERSION` (which can't carry `-alpha.1`
per `CFBundleShortVersionString` rules). The CHANGELOG extractor
falls back to the numeric version entry if no dedicated
pre-release entry exists. The resulting GitHub Release is marked
`--prerelease`.

**After tagging, upload to TestFlight** via `xcodebuild archive` +
`xcodebuild -exportArchive`. The `asc` CLI manages builds and
TestFlight groups from the terminal. See
[`docs/app-store-connect-workflow.md`](docs/app-store-connect-workflow.md)
for the full archive/upload/verify flow and one-time setup.

</details>

## When touching CI

<details>
<summary>iOS job structure, runner variance, test flakiness</summary>

**The iOS CI job is a single sequential job**, not split. Each
split test-job runner pays a full cold simulator boot (~30-60s),
XCUITest bootstrap (~30s), and simctl warmup (20-100s), which
drowns parallelism savings for our test suite. The current
workflow is `make build` + `make test-app`, intentional.

**DerivedData is cached across runs** via `actions/cache@v4`,
keyed on source hash + Xcode build version. Warm runs skip ~90% of
Swift compilation.

**CI runner variance is huge** — GitHub macos-15 runs vary by 2-3×
between builds. Don't optimize on noise. If a test run is
unusually slow, check whether it was a flaky simulator cascade
(see `docs/LEARNINGS.md` § "UI testing on iOS") before assuming
the workflow needs tuning.

**Don't use `xcodebuild test-without-building -xctestrun`** on CI
runners. Measured ~4 min slower per test pass than the
`-project`/`-scheme` path, for reasons that appear to be
CI-runner-specific. Works fine on dev Macs.

</details>

## When touching the simulator dev loop

<details>
<summary>Overlay install, keychain hygiene, test-mode vs real sessions</summary>

**To iterate on a live authenticated session without wiping auth:**

```bash
make sim-build                       # build only, no install
xcrun simctl install booted <.app>   # overlay install, keychain preserved
xcrun simctl launch booted com.indrasvat.jools
```

**Don't** `xcrun simctl uninstall` unless you're specifically
testing the onboarding flow. Uninstalling wipes the keychain and
forces re-authentication.

**Test mode vs real sessions:** `JOOLS_UI_TEST_MODE=1 +
JOOLS_UI_TEST_SCENARIO=running-session` loads seed fixtures. Fine
for XCUITest suites, wrong for "does this ship-able" verification.
For screenshots and live feature work, use a real authenticated
account.

**Keep the API key in `.env`** (gitignored) so re-authentication
after a necessary uninstall is programmatic:

```bash
awk -F= '/^JULES_API_KEY=/ {print $2}' .env \
  | tr -d '\n' \
  | xcrun simctl pbcopy booted
# then: axe tap --label "Paste from Clipboard" + --label "Connect"
```

Never echo the key to stdout.

</details>

## Session close-out checklist

At the end of any session that involved meaningful work:

- [ ] Did this session uncover a non-obvious lesson? → append to
      `docs/LEARNINGS.md` under the relevant section
- [ ] Does `CHANGELOG.md`'s `[Unreleased]` section need an entry?
      (any user-visible or architecturally-notable change does)
- [ ] Did the task touch `project.yml`? → `make generate`
- [ ] Did the task touch `scripts/ci-*`? → `shellcheck scripts/ci-*`
- [ ] Are there uncommitted `.ci-build*/` or `build-products.tar*`?
      → they're gitignored, ignore them
- [ ] Was the simulator left authenticated? → if yes, note it in
      the session summary so the next run doesn't start by
      wiping the keychain
